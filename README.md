# cs-230_TheGamingRoom
CS 230 Portfolio Reflection – Draw It or Lose It
Project Overview

The client for this project was The Gaming Room, which wanted to expand their existing Android game, Draw It or Lose It, into a web-based application that could support multiple platforms. The goal was to design a system that allows multiple teams and players to participate in a game while ensuring that all game and team names remain unique. The application needed to be scalable, organized, and accessible across different devices through a browser-based interface.

What I Did Well

One area I did particularly well was applying object-oriented design principles to structure the application. I clearly defined classes such as Game, Team, Player, and GameService, and used inheritance to reduce redundancy. I also successfully implemented the Singleton design pattern for the GameService class, which ensures that only one instance manages all game data. This helped maintain consistency and demonstrated my understanding of design patterns.

Helpful Design Process

Working through the software design document helped me better understand how to plan before coding. Breaking the system into components and defining relationships between classes made the implementation process more organized and efficient. It also helped me think ahead about scalability and how the application would function in a real-world environment, rather than just focusing on getting the code to work.

What I Would Improve

If I could revise part of my work, I would improve the system architecture section by adding a clearer visual or more detailed explanation of how components interact in a distributed environment. While the design was functionally correct, providing more detail about how the system would scale or be deployed in the cloud would strengthen the overall document.

Understanding User Needs

I interpreted the user’s needs by focusing on flexibility, scalability, and ease of use. The requirement for unique names and multiple teams influenced how I structured the system and managed data. Considering user needs is important because it ensures the software is both functional and intuitive. Without this focus, the application might meet technical requirements but fail to provide a good user experience.

Approach to Software Design

My approach to designing this software involved analyzing requirements, breaking the system into smaller components, and applying object-oriented principles to create a modular design. In the future, I would continue using strategies such as UML diagrams, design patterns, and iterative development to improve both planning and implementation. I would also incorporate more testing and feedback earlier in the process to refine the design before coding.
